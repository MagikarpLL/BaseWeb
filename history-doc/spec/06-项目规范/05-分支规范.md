# 分支规范

> 本文档定义项目的 Git 分支策略和工作流程。
> 详见编写规范: [CONVENTIONS.md](../../CONVENTIONS.md)

---

## 1. 分支类型

| 分支 | 命名 | 说明 | 保护 |
|------|------|------|------|
| 主分支 | `main` | 稳定版本，生产环境 | ✅ 必须 PR 合并 |
| 开发分支 | `develop` | 开发分支，集成所有 feature | ✅ 必须 PR 合并 |
| 功能分支 | `feature/{task-id}-{描述}` | 新功能开发 | - |
| 修复分支 | `fix/{task-id}-{描述}` | Bug 修复 | - |
| 发布分支 | `release/v{x.y.z}` | 发布版本 | ✅ 必须 PR 合并 |

---

## 2. 分支命名

### 2.1 功能分支

```
feature/{迭代}-{任务编号}-{简短描述}
```

| 示例 | 说明 |
|------|------|
| `feature/M1-01-后端项目初始化` | M1 迭代，任务 01 |
| `feature/M1-06-JWT认证` | M1 迭代，任务 06 |
| `feature/M2-07-关于页API` | M2 迭代，任务 07 |

### 2.2 修复分支

```
fix/{issue-id}-{简短描述}
```

| 示例 | 说明 |
|------|------|
| `fix/123-登录页面样式错乱` | 修复 issue #123 |
| `fix/456-分页组件边界错误` | 修复 issue #456 |

### 2.3 发布分支

```
release/v{x.y.z}
```

| 示例 | 说明 |
|------|------|
| `release/v1.0.0` | v1.0.0 正式版 |
| `release/v1.1.0` | v1.1.0 版本 |

## 3. 工作流程

### 3.1 功能开发流程

```
develop
  ↑
  │  merge (PR)
  │
feature/M1-01-xxx ──→ 创建 PR ──→ Review ──→ Merge ──→ 删除分支
```

**步骤：**
1. 从 `develop` 创建功能分支
2. 在分支上开发功能
3. 提交代码，确保通过 CI 检查
4. 创建 Pull Request 到 `develop`
5. 代码 Review 后合并
6. 删除功能分支

### 3.2 发布流程

```
main (生产)
  ↑
  │  merge (PR + Tag)
  │
release/v1.0.0
  ↑
  │  merge (PR)
  │
develop
  ↑
  │  merge (PR)
  │
feature/xxx
```

**步骤：**
1. 从 `develop` 创建发布分支 `release/v{x.y.z}`
2. 在发布分支上修复最后问题
3. 测试通过后，合并到 `main` 并打 Tag
4. 同时合并回 `develop`

### 3.3 热修复流程

```
main (生产)
  ↑
  │  merge (PR + Tag)
  │
hotfix/v1.0.1 ──→ 创建 PR ──→ 快速 Review ──→ Merge
    ↑
    │  从 main 创建
    │
main
```

**步骤：**
1. 从 `main` 创建热修复分支
2. 修复并测试
3. 同时合并到 `main`（打 Tag）和 `develop`

## 4. Pull Request 规范

### 4.1 PR 标题格式

```
{type}({scope}): {简短描述} - Closes #{task-id}
```

| 示例 | 说明 |
|------|------|
| `feat(backend): 添加 JWT 认证功能 - Closes #M1-06` | 新功能 |
| `fix(frontend): 修复博客列表分页 bug - Closes #M1-11` | Bug 修复 |
| `docs(all): 更新 README - Closes #M1-01` | 文档更新 |

### 4.2 PR 描述模板

```markdown
## 任务编号
{task-id}

## 涉及 Requirement
{requirement 文件}

## 涉及 Spec
{spec 文件}

## 变更内容
- 变更 1
- 变更 2

## 验证方式
- [ ] 通过 CI 检查
- [ ] 人工 Review
```

## 5. 分支状态

| 状态 | 标识 | 说明 |
|------|------|------|
| 活跃 | - | 正在开发中 |
| 待 Review | WIP | 有未完成的变更 |
| 待合并 | Ready | 已完成，等待合并 |
| 已合并 | Merged | 已合并到目标分支 |
| 已关闭 | Closed | 不需要合并 |

## 6. 常用 Git 命令

```bash
# 创建功能分支
git checkout develop
git pull
git checkout -b feature/M1-01-后端项目初始化

# 提交变更
git add .
git commit -m "feat(backend): 后端项目初始化 - Closes #M1-01"

# 推送分支
git push -u origin feature/M1-01-后端项目初始化

# 创建 PR（使用 gh CLI）
gh pr create --base develop --head feature/M1-01-后端项目初始化 --title "feat(backend): 后端项目初始化 - Closes #M1-01"

# 合并 PR
gh pr merge --admin --squash

# 删除已合并的分支
git branch -d feature/M1-01-后端项目初始化
git push origin --delete feature/M1-01-后端项目初始化
```

## 7. 注意事项

- **禁止**直接在 `main` 和 `develop` 分支上提交代码
- **禁止**在功能分支上开发跨任务功能（保持单一职责）
- 每次提交应保持原子性（一个提交一个功能点）
- PR 合并前必须通过 CI 检查
