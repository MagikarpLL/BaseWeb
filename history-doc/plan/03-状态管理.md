# 03-状态管理规范

> 本文档定义 `.harness/state.json` 的结构、字段含义和流转规则。
> 是 Harness 工程的核心状态文件，支持断点继续和定时触发恢复。
> 详见编写规范: [CONVENTIONS.md](../CONVENTIONS.md)

---

## 1. 状态文件位置

```
{项目根目录}/
  .harness/
    state.json          # 主状态文件
    state.json.example  # 模板文件（仅供参考）
```

**注意**：`.harness/` 目录应添加到 `.gitignore` 中，不提交到版本控制。

---

## 2. 状态文件结构

```json
{
  "version": "1.0",
  "phase": "idle|requirement|spec|implement|verify|paused|completed",
  "iteration": "M1|M2|M3|M4",
  "session": {
    "main_session_id": null,
    "req_session_id": null,
    "spec_session_id": null,
    "impl_session_id": null
  },
  "checkpoints": {
    "requirement": { "doc": null, "confirmed": false, "step": null, "timestamp": null },
    "spec": { "requirement_doc": null, "spec_doc": null, "confirmed": false, "step": null, "timestamp": null },
    "implement": { "iteration": null, "task_id": null, "step": null, "attempt": 0, "max_attempts": 5, "timestamp": null },
    "verify": { "spec": null, "passed": false, "timestamp": null }
  },
  "tasks": {
    "queue": [],
    "completed": [],
    "failed": []
  },
  "pending_docs": [],
  "confirmed_docs": [],
  "messages": [],
  "last_activity": null
}
```

## 3. 字段说明

### 3.1 顶层字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `version` | string | 状态文件版本，用于兼容性 |
| `phase` | string | 当前工程阶段 |
| `iteration` | string | 当前迭代编号 |
| `session` | object | 各技能的 session ID，用于保持连续性 |
| `checkpoints` | object | 各阶段的检查点，支持断点继续 |
| `tasks` | object | 任务队列状态（仅 implement 阶段使用） |
| `pending_docs` | array | 待确认的文档列表 |
| `confirmed_docs` | array | 已确认的文档列表 |
| `messages` | array | 工程日志消息 |
| `last_activity` | string | 最后活动时间（ISO 8601） |

### 3.2 phase 阶段定义

| 阶段 | 说明 | 可转换到 |
|------|------|----------|
| `idle` | 空闲状态 | `requirement`、`spec`、`implement` |
| `requirement` | 需求确认阶段 | `spec`（需求确认后）、`paused` |
| `spec` | 规格生成阶段 | `implement`（规格确认后）、`paused` |
| `implement` | 实现阶段 | `verify`、`paused`、`completed` |
| `verify` | 验证阶段 | `implement`（验证失败）、`completed`（验证通过） |
| `paused` | 暂停状态 | 任何阶段（通过 resume） |
| `completed` | 完成状态 | - |

### 3.3 session 字段

| 字段 | 说明 |
|------|------|
| `main_session_id` | 主技能 harness 的 session ID |
| `req_session_id` | harness-req 的 session ID |
| `spec_session_id` | harness-spec 的 session ID |
| `impl_session_id` | harness-impl 的 session ID |

### 3.4 checkpoints 字段

#### requirement 检查点

| 字段 | 类型 | 说明 |
|------|------|------|
| `doc` | string | 当前需求文档路径 |
| `confirmed` | boolean | 是否已确认 |
| `step` | string | 当前步骤：`clarify`、`draft`、`request`、`confirm`、`done` |
| `timestamp` | string | 最后更新时间 |

#### spec 检查点

| 字段 | 类型 | 说明 |
|------|------|------|
| `requirement_doc` | string | 关联的需求文档路径 |
| `spec_doc` | string | 生成的规格文档路径 |
| `confirmed` | boolean | 是否已确认 |
| `step` | string | 当前步骤：`analyze`、`generate`、`request`、`confirm`、`done` |
| `timestamp` | string | 最后更新时间 |

#### implement 检查点

| 字段 | 类型 | 说明 |
|------|------|------|
| `iteration` | string | 当前迭代编号 |
| `task_id` | string | 当前任务编号 |
| `step` | string | 当前步骤：`scan`、`implement`、`verify`、`commit`、`done`、`paused` |
| `attempt` | number | 当前修复尝试次数 |
| `max_attempts` | number | 最大修复尝试次数（默认 5） |
| `timestamp` | string | 最后更新时间 |

#### verify 检查点

| 字段 | 类型 | 说明 |
|------|------|------|
| `spec` | string | 验证对应的规格文档 |
| `passed` | boolean | 验证是否通过 |
| `timestamp` | string | 最后更新时间 |

### 3.5 tasks 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `queue` | array | 待实现任务队列 |
| `completed` | array | 已完成任务列表 |
| `failed` | array | 失败任务列表 |

## 4. 阶段流转规则

```
                    ┌─────────────┐
                    │    idle    │
                    └─────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                    requirement 阶段                       │
│  clarify → draft → request → confirm → done              │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                    spec 阶段                             │
│  analyze → generate → request → confirm → done          │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                   implement 阶段                         │
│  scan → next → implement → verify → commit → next       │
│                              │              │            │
│                              ▼              ▼            │
│                         [auto-repair]   [下一任务]        │
│                              │              │            │
│                              ▼              ▼            │
│                         [paused]       [done]            │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│                   verify 阶段                            │
│  start → check → report                                 │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
                    ┌─────────────┐
                    │ completed  │
                    └─────────────┘
```

## 5. 断点继续规则

### 5.1 定时触发恢复流程

```
定时触发 harness
  │
  ├─ 读取 .harness/state.json
  │
  ├─ phase = "idle"
  │    └─ 等待人工触发
  │
  ├─ phase = "requirement"
  │    ├─ checkpoint.step = "clarify" → 调用 harness-req clarify 继续
  │    ├─ checkpoint.step = "draft" → 调用 harness-req draft 继续
  │    ├─ checkpoint.step = "request" → 提示用户确认
  │    └─ checkpoint.step = "confirm" → 调用 harness-req confirm
  │
  ├─ phase = "spec"
  │    ├─ checkpoint.step = "analyze" → 调用 harness-spec analyze
  │    ├─ checkpoint.step = "generate" → 调用 harness-spec generate
  │    ├─ checkpoint.step = "request" → 提示用户确认
  │    └─ checkpoint.step = "confirm" → 调用 harness-spec confirm
  │
  ├─ phase = "implement"
  │    ├─ checkpoint.step = "paused"
  │    │    ├─ attempt >= max_attempts → 报告失败，停止
  │    │    └─ attempt < max_attempts → 调用 harness-impl auto-repair
  │    │
  │    ├─ checkpoint.step = "implement" → 调用 harness-impl verify
  │    ├─ checkpoint.step = "verify" → 调用 harness-impl commit 或 auto-repair
  │    └─ checkpoint.step = "commit" → 调用 harness-impl next
  │
  └─ phase = "verify"
       └─ 调用 harness-impl verify 继续
```

### 5.2 自动修复规则

| 条件 | 动作 |
|------|------|
| 验证失败且 `attempt < max_attempts` | 自动调用 `auto-repair`，递增 `attempt` |
| 验证失败且 `attempt >= max_attempts` | 将任务移入 `failed`，暂停，报告 |
| 修复后验证通过 | 继续 `commit` 流程 |
| 用户主动暂停 | 更新 `phase` = "paused"，保存原因 |

## 6. 消息格式

每条消息格式：`[时间戳] [类型] 内容`

| 类型 | 说明 |
|------|------|
| `[Q]` | 提问 |
| `[DRAFT]` | 起草文档 |
| `[CONFIRMED]` | 确认文档 |
| `[CHECKPOINT]` | 检查点 |
| `[COMPLETE]` | 完成任务 |
| `[PASS]` | 验证通过 |
| `[FAIL]` | 验证失败 |
| `[REPAIR]` | 修复尝试 |
| `[PAUSE]` | 暂停 |
| `[RESUME]` | 恢复 |

## 9. 注意事项

- **状态文件不提交**：`.harness/` 目录应添加到 `.gitignore`
- **每次操作前读取**：所有技能在修改状态前必须先读取当前状态
- **每次操作后写回**：状态变更后必须立即写回文件
- **保持 session 连续**：子技能调用后更新对应的 session_id
