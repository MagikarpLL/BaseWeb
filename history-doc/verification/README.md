# 验收测试文档 (Verification)

> 本目录包含各迭代任务的验收测试文档。
> 由 AI 自动启动 agent 执行验证，用户只负责定义标准。
> **核心原则**：Verification 中的测试必须是**实际可执行的命令**，而非静态代码检查。

## 文档模板

> **重要**：测试用例必须包含**执行命令**，格式如下：

```markdown
# {任务编号} - {任务名称}

> 对应任务: [01-迭代计划.md](../plan/01-迭代计划.md#任务-{task-id})
> 涉及 Spec: [{Spec 文件}](../spec/{路径})

## 验证目标

描述验收的核心目标。

## 测试用例

### TC-01: {测试用例名称}

**执行命令**:
```bash
cd {frontend|backend} && {实际命令}
```
例如：
- `cd frontend && npm run build`
- `cd backend && mvn test -Dtest=*MapperTest`
- `cd frontend && npm run test:e2e -- --grep "LanguageSwitch"`

**预期结果**:
- 退出码 = 0
- 输出包含成功标志（如 `BUILD SUCCESS`、`3 passed`）

### TC-02: {测试用例名称}

...（同上格式）

## 验证报告（由 AI 填写）

**验证时间**：{YYYY-MM-DD HH:mm:ss}
**验证结果**：✅ PASSED / ❌ FAILED

### TC-01: {测试用例名称}
- **命令**: `{实际执行的命令}`
- **退出码**: {0|非0}
- **运行时间**: {Xs}
- **状态**: ✅ PASSED / ❌ FAILED
- **输出摘要**: {命令输出的关键部分}

## 验收标准

- [ ] TC-01 通过
- [ ] TC-02 通过
- [ ] 所有测试通过
```

---

## AI 验证流程

1. AI 读取 `docs/verification/{iteration}/{task_id}.md`
2. **解析测试用例中的"执行命令"字段**
3. **实际执行命令**（不是只读代码检查）
4. 捕获命令输出和退出码
5. 判定 pass/fail（退出码 = 0 为通过）
6. 生成验证报告（包含实际命令输出）
7. 如有失败，执行自动修复（最多 5 次）
8. 更新文档中的验收标准 checkbox

## 与 Spec 的区别

| 文档 | 回答的问题 | 编写者 |
|------|-----------|--------|
| Spec | 怎么做（技术实现） | AI 与用户讨论 |
| Verification | 如何验证（**实际运行测试**） | 用户定义命令，AI 执行 |

---

## 注意事项

- **执行命令是必须的**：每个测试用例必须包含可执行的命令
- **不能只做静态检查**：读取代码检查结构不算验证，必须实际运行
- **验证判定标准**：退出码 = 0 为通过，非 0 为失败
- **失败时提供实际输出**：错误信息应包含实际命令的输出，便于调试
- **测试用例应覆盖正向和负向场景**
- 复杂场景可附加截图或日志示例
