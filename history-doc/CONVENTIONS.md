# 文档编写规范 (CONVENTIONS)

> 本文档定义 `docs/` 目录下所有文档的编写规范，确保格式统一、术语一致、可追溯性完整。
> 所有新建文档必须遵循本规范。

---

## 1. 文档三层结构

| 层次 | 目录 | 回答的问题 | 内容范围 |
|------|------|-----------|----------|
| Requirement | `docs/requirements/` | 做什么？ | 用户故事、页面布局、交互行为、验收标准 |
| Spec | `docs/spec/` | 怎么做？ | API 端点、JSON 结构、数据库 DDL、代码实现 |
| Plan | `docs/plan/` | 何时做？ | 迭代划分、任务拆分、优先级、依赖关系 |

### 禁止越界

- Requirement 中**不得**包含技术实现细节（如 JSON 结构、SQL、API 端点、代码片段）
- Spec 中**不得**重复业务需求描述（如"为什么做"、"用户价值"）
- Plan 中**不得**引入新的需求或规格，只做编排

### 越界处理

发现越界内容时：
1. 将越界内容移至正确的层次
2. 在原位置添加引用链接：`> xxx 的详细定义见 Spec: [文件名](相对路径)`
3. 在追溯矩阵中记录变更

## 2. 文件命名规范

### Requirement 文件

```
docs/requirements/{模块编号}-{模块名称}/{模块编号}-{子编号}-{页面/功能名称}.md
```

| 规则 | 示例 |
|------|------|
| 模块编号两位数，左补零 | `02-个人网站展示模块` |
| 子编号两位数，左补零 | `02-01-首页.md` |
| 使用中文命名 | `03-博客管理模块` |
| 模块编号可跳号（保留扩展空间） | 04→07→08→09→10（05、06 预留） |

### Spec 文件

```
docs/spec/{两位编号}-{名称}.md
docs/spec/{两位编号}-{名称}/{两位子编号}-{名称}.md
```

| 规则 | 示例 |
|------|------|
| 两位编号，左补零 | `01-技术架构.md` |
| 使用中文命名 | `03-数据库设计.md` |
| 子目录用两位编号 | `06-项目规范/01-后端项目结构.md` |

### Plan 文件

```
docs/plan/{两位编号}-{名称}.md
```

## 3. 章节编号规范

### 格式

```
{模块编号}.{子编号}.{序号}
```

| 示例 | 说明 |
|------|------|
| `7.1.1` | 模块 07，子模块 01，第 1 节 |
| `3.3.2` | 模块 03，子模块 03，第 2 节 |

### 标准章节结构

每个 Requirement 文档**必须**包含以下章节（按顺序）：

| 序号 | 章节标题 | 必选 | 说明 |
|------|----------|------|------|
| x.x.1 | 功能概述 | ✅ | 一段话描述本模块做什么 |
| x.x.2 | 页面布局 | ⬜ | 前台页面必须有，纯后端功能可省略 |
| x.x.3 | 功能详情 | ✅ | 字段表格、规则说明 |
| x.x.4 | SEO 元数据 | ⬜ | 仅前台公开页面需要 |
| x.x.5 | 用户交互 | ⬜ | 仅页面类需求需要 |
| x.x.6 | 验收标准 | ✅ | 必须有，使用 checkbox 列表 |

每个 Spec 文档**必须**包含以下章节（按顺序）：

| 序号 | 章节标题 | 必选 | 说明 |
|------|----------|------|------|
| 1 | 概述/设计原则 | ✅ | 本 Spec 的核心决策和原则 |
| 2..N | 具体规格 | ✅ | 按 Spec 主题组织 |
| N+1 | 验收标准 | ✅ | 必须有 |

## 4. 交叉引用规范

### 引用格式

| 引用目标 | 格式 | 示例 |
|----------|------|------|
| Requirement 文件 | `[显示名](相对路径)` | `[03-03-博客编辑](../03-博客管理模块/03-03-博客编辑.md)` |
| Spec 文件 | `[显示名](相对路径)` | `[02-API契约](../../spec/02-API契约.md)` |
| Plan 文件 | `[显示名](相对路径)` | `[01-迭代计划](../plan/01-迭代计划.md)` |
| 章节内引用 | `> xxx 的详细定义见 Spec: [文件名](相对路径)` | 引用行前用 `>` 标记 |

### 引用原则

- **向上引用**（Requirement 引用 Spec）：在字段描述旁标注数据来源
- **向下引用**（Spec 引用 Requirement）：在 Spec 文件头部标注 `对应 Requirement: [文件名]`
- **平级引用**（Spec 引用 Spec）：在相关章节添加链接
- **Plan 引用**：Plan 中的每个任务必须标注"涉及 Requirement"和"涉及 Spec"

### 禁止硬编码重复

如果某个定义已存在于其他文档，**不得**在当前文档中重新写一遍。必须使用引用。

## 5. 表格格式规范

### 项目内统一使用 `||` 表格语法

```markdown
|| 列1 | 列2 |
||------|------|
|| 值1 | 值2 |
```

**不使用** `|` 开头的标准 Markdown 表格（因为项目中所有已有文档均使用 `||` 格式，保持一致）。

### 表格列顺序

| 表格类型 | 必选列 | 推荐列顺序 |
|----------|--------|-----------|
| 字段定义表 | 字段、说明 | 字段 → 类型 → 必填 → 说明 |
| API 接口表 | 接口、方法、说明 | 接口 → 方法 → 说明 → 认证 |
| 验收标准 | checkbox 列表 | - [ ] 描述 |

## 6. 状态标记规范

### 验收标准使用 checkbox

```markdown
- [ ] 功能正常工作
- [ ] 边界场景处理正确
```

### 文档状态

| 状态 | 标记 | 说明 |
|------|------|------|
| 草稿 | 无标记（默认） | 正在编写中 |
| 已评审 | 在文档头部添加 `> 状态: 已评审 ✓` | 经过评审确认 |
| 已实现 | 在文档头部添加 `> 状态: 已实现 ✅` | 代码已实现 |
| 已废弃 | 在文档头部添加 `> 状态: 已废弃 ❌` | 不再使用 |

---

## 7. 页面布局图规范

### 格式

使用 ASCII 框线图，统一使用以下字符：

```
┌ ─ ┐ │ └ ┘ ├ ┤ ┬ ┴ ┼
```

### 规则

- 外框使用双线 `┌─┐`
- 内部分区使用单线 `├─┤`
- 每个区域标注名称
- 宽度保持一致（约 65 字符）

## 8. 新建文档检查清单

新建任何文档前，确认以下事项：

- [ ] 确认文档属于 Requirement / Spec / Plan 哪一层
- [ ] 确认文件命名符合编号规范
- [ ] 确认章节结构符合标准模板
- [ ] 确认不包含越界内容（Requirement 无技术细节，Spec 无业务描述）
- [ ] 确认交叉引用指向正确、路径有效
- [ ] 确认验收标准完整且可测试
- [ ] 更新对应目录的 README.md 索引
- [ ] 在 `plan/02-验收追溯矩阵.md` 中添加追溯关系
