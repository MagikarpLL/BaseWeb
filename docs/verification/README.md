# 验收测试文档 (Verification)

> 本目录包含各迭代任务的验收测试文档。
> 由 AI 自动启动 agent 执行验证，用户只负责定义标准。

---

## 目录结构

```
docs/verification/
  M1/                      # M1 迭代验证文档
    M1-01-后端项目初始化.md
    M1-02-前端项目初始化.md
    M1-03-Docker环境.md
    ...
  M2/                      # M2 迭代验证文档
  M3/                      # M3 迭代验证文档
  M4/                      # M4 迭代验证文档
```

---

## 文档命名

```
{迭代}-{任务编号}-{任务名称}.md
```

| 示例 | 说明 |
|------|------|
| `M1-01-后端项目初始化.md` | M1 迭代，任务 01 |
| `M2-05-网站配置API.md` | M2 迭代，任务 05 |

---

## 文档模板

```markdown
# {任务编号} - {任务名称}

> 对应任务: [01-迭代计划.md](../plan/01-迭代计划.md#任务-{task-id})
> 涉及 Spec: [{Spec 文件}](../spec/{路径})

## 验证目标

描述验收的核心目标。

## 测试用例

### TC-01: {测试用例名称}

**前置条件：**
- 条件 1
- 条件 2

**测试步骤：**
1. 步骤 1
2. 步骤 2
3. 步骤 3

**预期结果：**
- 结果 1
- 结果 2

### TC-02: {测试用例名称}
...

## 验证方法

| 方法 | 说明 |
|------|------|
| 编译检查 | `mvn compile` 无错误 |
| 单元测试 | `mvn test` 通过 |
| 端到端 | Playwright 自动化测试 |

## 验收标准

- [ ] TC-01 通过
- [ ] TC-02 通过
- [ ] 所有测试通过
```

---

## AI 验证流程

1. AI 读取 `docs/verification/{iteration}/{task_id}.md`
2. 按照测试用例执行验证
3. 记录每条 TC 的通过/失败状态
4. 生成验证报告
5. 如有失败，执行自动修复（最多 5 次）
6. 更新文档中的验收标准 checkbox

---

## 与 Spec 的区别

| 文档 | 回答的问题 | 编写者 |
|------|-----------|--------|
| Spec | 怎么做（技术实现） | AI 与用户讨论 |
| Verification | 如何验证（测试用例） | 用户定义，AI 执行 |

---

## 注意事项

- 验收标准必须可测试（不能是主观判断）
- 测试用例应覆盖正向和负向场景
- 复杂场景可附加截图或日志示例
